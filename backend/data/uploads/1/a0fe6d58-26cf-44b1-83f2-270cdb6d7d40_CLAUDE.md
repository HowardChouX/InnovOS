## Testing and Running

### Python Homeworks (hw01-hw06) and Discussion Exercises

Run tests using the `ok` testing framework (a custom UC Berkeley testing tool):

```bash
# Navigate to the assignment directory first
cd hw/hw01/

# Run all tests
python3 ok

# Run specific question (e.g., question 1)
python3 ok -q q1

# Run tests with verbose output (shows test names and results)
python3 ok -v

# Unlock tests (interactive - shows hidden test cases after solving)
python3 ok -u

# Run tests with local mode (for offline development)
python3 ok --local

# Check syntax without running tests
python3 ok --check
```

**Important Notes:**
- The `ok` file in each directory is a self-contained executable (actually a zip file)
- Tests are defined in `.ok` config files (e.g., `hw01.ok`)
- Test results are stored in `.ok_storage.*` files
- Use `--local` flag when offline or if server connection fails
- Syntax checking uses `construct_check.py` to validate code structure constraints

### Scheme Homeworks (hw07-hw10) and Scheme Labs

Launch the web-based Scheme editor (a full Scheme interpreter with visualization):

```bash
# From the assignment directory (e.g., hw/hw07/)
# Launch editor (opens browser automatically)
python3 editor

# Launch without opening browser
python3 editor -nb

# Specify custom port
python3 editor -p 31415

# Test specific Scheme files
python3 editor -f hw07.scm

# Reformat Scheme code (auto-indent)
python3 editor -r hw07.scm

# Run without dotted list support (use -n flag to disable)
python3 editor -n

# Display help with all options
python3 editor -h
```

**Editor Architecture:**
- The editor is a local web server (`local_server.py`) running on port 31415
- It connects to a Scheme interpreter backend with visualization
- Supports Racket-style Scheme syntax (`#lang racket`)
- Provides real-time evaluation and visualization of Scheme code

## Code Architecture

### Python Homeworks Structure

Each Python assignment directory contains:
- `hwXX.py` - Main assignment file with student code
- `construct_check.py` - AST-based validation to disallow certain Python constructs (e.g., loops, specific operators) in specific functions
- `hwXX.ok` - Test configuration for the `ok` framework
- `ok` - Self-contained testing executable
- `tests/` - Test module directory (usually just `__init__.py`)

**Construct Checking System:**
- `construct_check.py` uses Python's AST module to parse and validate code
- Functions often have `*_syntax_check()` companion functions that verify code structure
- Validates that students don't use prohibited constructs (loops, certain operators, etc.)
- Special node name 'Recursion' checks for overtly recursive calls
- Returns `True` if no prohibited constructs are found

**Common Python Patterns:**
- Doctests are embedded in function docstrings for testing
- Functions return values (not print) unless specified
- Many assignments use functional programming patterns (higher-order functions, recursion)
- The `Link` class (`hw/hw06/Link.py`) implements a custom linked list data structure used throughout the course

### Scheme Editor Architecture (hw07-hw09, labs, projects)

The Scheme editor is a full Scheme interpreter with a web-based visualization interface, replicated in each assignment directory:

**Core Components (in `editor/` directory):**
- `__main__.py` - CLI argument handling and server launch
- `lexer.py` - Tokenizes Scheme source code into tokens
- `special_forms.py` - Handles Scheme special forms (define, lambda, if, begin, quote, etc.)
- `evaluate_apply.py` - Evaluation engine with Frame-based environment model
- `environment.py` - Global environment and special form registration
- `datamodel.py` - Core data structures (Symbol, Expression, Pair, Nil, Promise, etc.)
- `execution.py` - Main execution loop
- `local_server.py` - Web server for the editor interface (port 31415)
- `visualizing.py` - Visualization logic for the web UI
- `builtins.scm` - Built-in Scheme procedures
- `formatter.py` - Scheme code formatting and pretty-printing

**Key Concepts:**
- **Frame-based scoping**: Environments are chains of Frames with parent references
- **Tail call optimization**: The evaluator supports tail context for proper tail recursion
- **Thunk-based lazy evaluation**: Used for promises and delayed evaluation
- **Logging system**: Comprehensive logging for visualization and debugging
- **Dotted lists**: Supported by default (disable with `-n` flag)

**Entry Point**: `editor/__main__.py` handles CLI arguments and launches the local server. The server provides a web interface for editing and visualizing Scheme code execution.

## Common Patterns and Development Workflow

### Python Development

1. **Code Structure Validation**:
   - Functions often have `*_syntax_check()` companion functions
   - Use `construct_check.py` to validate prohibited constructs
   - Syntax checks run automatically when tests are executed

2. **Testing Workflow**:
   ```bash
   # 1. Implement a function in hwXX.py
   # 2. Run tests for that specific question
   python3 ok -q q1

   # 3. If tests fail, fix implementation and retest
   # 4. Unlock hidden tests after solving
   python3 ok -u

   # 5. Run all tests to ensure nothing is broken
   python3 ok
   ```

3. **Debugging Tools**:
   - Use `print()` statements for debugging (remove before submission)
   - Doctests in docstrings provide immediate feedback
   - The `ok` framework shows which specific tests fail

### Scheme Development

1. **Scheme Code Patterns**:
   - Uses Racket-style Scheme syntax (`#lang racket`)
   - Common patterns: `define`, `lambda`, `cond`, `let`, `begin`, `quote`
   - Lists are fundamental data structure (`cons`, `car`, `cdr`)
   - Recursion is preferred over iteration

2. **Editor Workflow**:
   ```bash
   # 1. Launch editor with your Scheme file
   python3 editor -f hw07.scm

   # 2. Edit code in the web interface
   # 3. Use "Run" button to evaluate expressions
   # 4. Visualize execution steps in the interface
   # 5. Reformat code for consistent style
   python3 editor -r hw07.scm
   ```

3. **Common Scheme Functions**:
   - List manipulation: `map`, `filter`, `foldr`, `foldl`
   - Number operations: `+`, `-`, `*`, `/`, `modulo`, `expt`
   - Predicates: `null?`, `pair?`, `list?`, `number?`, `symbol?`
   - Control flow: `if`, `cond`, `and`, `or`

### Project Development

**Hog Project Specifics**:
- Strategy functions must return integer 0-10 (dice count)
- Use `make_test_dice()` for deterministic testing
- Implement `final_strategy()` as the optimal strategy
- GUI shows game visualization and strategy performance

## Troubleshooting

### Common Issues

1. **`ok` Framework Issues**:
   ```
   # If connection fails, use local mode
   python3 ok --local

   # If tests hang, check for infinite loops
   # Use Ctrl+C to interrupt, then fix code
   ```

2. **Scheme Editor Issues**:
   ```
   # If port 31415 is in use
   python3 editor -p 31416

   # If browser doesn't open
   python3 editor -nb
   # Then manually go to http://localhost:31415
   ```

3. **Import Errors**:
   - Ensure you're in the correct directory
   - Python assignments: `cd hw/hw01/`
   - Scheme assignments: `cd hw/hw07/`

4. **Syntax Check Failures**:
   - Review `construct_check.py` error messages
   - Ensure you're not using prohibited constructs (loops, certain operators, etc.)
   - Check function names match exactly
   - Verify code structure matches assignment requirements
